# V9 Design — Revise 모드 (주기적 재결정형 adaptive)

상태: **설계 단계, 미구현.** 코드 변경 없음. 2026-07-21 대화에서 확정된 방향을 문서화.

## 1. 배경 및 목적

`완익 사전결정 구조의 한계`(`V9 Issue - 완익 사전결정 구조.md`) 대응으로 이 세션에서 두 갈래를 검토했다.

1. **rl 모드**(Discrete(3), 2026-07-20 재설계, 기존): 매 1분마다 Hold/Enter/Close 중 하나를 판단하는 완전 밀집형. 실측 결과 과매매 스팸 ↔ 완전 무행동 사이를 오가는 불안정이 반복됐고(2026-07-21 두 차례 재학습 모두), valid에서 봐줄 만했던 유일한 체크포인트(0720-2340_best)도 test에서 완전히 무너짐(복리 0.53~0.56배). 원인 후보로 explore_bonus/ent_coef 스케줄 타이밍을 지목해 조정 중이나, 근본적으로 "1분 단위 밀집 보상 + 3지 이산 행동"이라는 조합 자체가 adaptive(semi-MDP)보다 크레딧 할당이 훨씬 어려운 구조라는 의심이 있음.
2. **rl 모드 + `decision_stride` 확대** (예: `--stride 10`): 기존 rl 모드 코드 그대로, 정책 호출 빈도만 낮추는 방법. 구현 비용은 0에 가깝지만, adaptive가 가진 연속 파라미터(레버리지/손절폭/반익·완익 목표) 표현력을 못 씀 — **rl 모드의 하위호환**에 불과하다는 지적(2026-07-21 사용자)이 정확함.

**Revise 모드**는 이 절충안이다: **adaptive의 `Box(-1,1)^6` 연속 행동공간의 표현력은 유지하면서, 진입 시점에만 파라미터를 박제하는 대신 일정 주기(예: 10분)마다 그 파라미터를 다시 판단**하게 한다. rl 모드보다 결정 빈도가 훨씬 낮아(1분→N분) semi-MDP에 가까운 학습 안정성을 상당 부분 회복하면서도, "보유 중 새 정보로 판단을 갱신"한다는 원래 목적은 달성한다.

## 2. 명명

| 항목 | 값 |
|---|---|
| `exit_mode` 값 | `"revise"` |
| 산출물 폴더 태그 | `v9_revise` (`models/v9_revise/`, `logs/v9_revise/`) — `mode_subdir()`에 분기 추가 필요 |
| run_name mode_tag | `"revise"` (예: `v9_revise_seed0_MMDD-HHMM`) |
| 한글 지칭 | "재조정형 모드" 또는 "주기 재결정 모드" |

기존 세 모드(`rule`/`adaptive`/`rl`)와 나란히 네 번째 `exit_mode`가 된다. 관측/행동 공간이 셋과 전부 달라 체크포인트 호환은 없음(기존 관례와 동일).

## 3. 관측 공간

adaptive의 기존 18차원 + **포지션 상태 4칸**(rl 모드에서 쓰던 것과 동일 개념, 무포지션일 땐 전부 0 또는 중립값으로 채움):

| # | 이름 | 설명 |
|---|---|---|
| 1 | `pos_dir` | 1(롱)/0(무포지션)/-1(숏) |
| 2 | `unrealized_pnl` (정규화) | 현재 미실현손익, adaptive 관측 스케일에 맞춰 ~[-1,1] 재스케일 (rl 모드 2026-07-20 수정과 동일 원칙) |
| 3 | `hold_elapsed` (정규화) | 진입 후 경과 시간 |
| 4 | `dist_to_sl_tp` 등 | 현재 손절/반익/완익 목표까지의 남은 거리 — 구체 정의는 구현 시 확정 |

총 22차원 (rl 모드와 동일 차원수이나 값의 의미는 다름).

## 4. 행동 공간: `Box(-1,1)^6`, 상태-조건부 재해석

축 구성은 adaptive와 동일하되, **재결정 시점의 포지션 상태에 따라 같은 축의 의미가 달라진다** (rl 모드의 "문 앞의 로봇" 패턴을 이산이 아니라 연속 액션에 적용).

| 축 | 무포지션일 때 (진입 결정) | 보유중일 때 (재결정) |
|---|---|---|
| a[0] | >0: 진입 / ≤0: Skip | >0: 유지(Hold) / ≤0: 지금 청산(Close) |
| a[1] 레버리지 | 진입 레버리지 결정 (`dynamic_leverage_ceiling` 등 기존 로직 그대로 적용) | **사용 안 함** — 진입 후 레버리지 재조정은 지원하지 않음(포지션 사이즈 변경은 거래소 실전 운용과 괴리 커짐) |
| a[2] 손절폭 (sl_multiplier) | 초기 손절 거리 결정 | 손절선 재조정 (트레일링 등) — 완화만 허용할지 강화도 허용할지는 구현 시 결정 |
| a[3] 반익 레벨 (tp_half) | 초기 반익 목표 | 반익 목표 갱신 |
| a[4] 완익 추가폭 (tp_full_extra) | 초기 완익 목표 | 완익 목표 갱신 |
| a[5] 반익 실행여부 (do_half) | 반익 계획 여부 | 아직 반익 미실행이면 지금 실행할지 |

## 5. 환경 로직 변경 (핵심 구현 과제)

가장 큰 작업은 **엔진을 "끊길 수 있게" 만드는 것**이다. 현재 `simulate_adaptive_exit`은 진입 시점부터 자연 청산까지 한 번에 계산해버리는 함수라 재사용 불가.

- 새 시뮬레이션 로직은 "다음 재결정 시점(`decision_stride`분 후)까지, 혹은 그 전에 손절/반익/완익 중 하나라도 먼저 도달하면 그 시점까지" 중 **더 이른 쪽**까지만 전진해야 한다.
- 재결정 시점에 도달했는데 아직 청산 조건에 안 걸렸으면, 포지션을 유지한 채로 `step()`을 반환하고 다음 행동을 정책에게 받는다.
- 이 사이 반익(half TP)이 실행됐는지, 손절/완익 목표가 이전 결정에서 바뀌었는지 등의 상태를 env 내부에 들고 있어야 함 — rl 모드의 `pos_dir`/`_open_position`/`_close_position` 패턴을 재사용/확장.
- 2026-07-21에 rl 모드에 적용한 "종료 시 강제청산 완화"(에피소드 끝나도 포지션 보유 중이면 자연 청산까지 연장) 원칙을 동일하게 적용해야 함 — 완전히 같은 이유(정책 판단과 무관한 임의 시점 손익 확정 방지).

## 6. 보상 설계 (미확정, 검토 필요)

- semi-MDP처럼 "청산 시 1회 실현손익"만 줄지, rl 모드처럼 재결정 시점마다 그 구간의 mark-to-market 변화량을 줄지 결정 필요.
- 후자가 자연스러워 보이지만(재결정 구간별로 크레딧이 분산됨), 이렇게 하면 rl 모드가 겪은 "크레딧 할당 어려움"을 정도는 약하게나마 다시 물려받는다는 트레이드오프가 있음 — 재결정 주기(N)를 늘릴수록 semi-MDP에 가까워지고 줄일수록 rl 모드에 가까워지는 스펙트럼으로 이해하는 게 정확함.
- explore_bonus 필요 여부도 재검토 대상(무포지션 상태에서 Enter를 유인해야 하는 콜드스타트 문제는 adaptive/rl과 동일하게 존재할 가능성 높음).

## 7. 알려진 리스크 / 불확실성

- **미검증 가설**: "결정 빈도를 낮추면 rl 모드의 불안정이 완화될 것"이라는 전제 자체가 아직 실측된 바 없음. rl 모드에서 `--stride`만 올려보는 실험(구현 비용 0)을 먼저 돌려서 이 가설을 저비용으로 먼저 확인해보는 것을 권장 — Revise 모드 구현에 들어가는 공수(오늘 rl 모드 재설계급) 대비 선행 검증 가치가 있음.
- 관측 4칸(포지션 상태)의 구체적 정규화 방식은 rl 모드 때와 동일한 시행착오(예: 미실현손익 스케일 불일치로 한 번 수정한 전례, 2026-07-20)를 반복할 수 있음 — 처음부터 다른 관측 스케일과 일치시키는 데 주의.
- 손절 재조정을 "완화" 방향도 허용할지(트레일링 스탑 강화만 허용하는 게 더 안전할 수 있음 — 완화를 허용하면 손실 확대 방향으로 정책이 악용할 리스크) 는 별도 논의 필요.

## 8. 구현 순서 제안

1. `--stride` 확대 실험(rl 모드, 비용 0)으로 "결정 빈도 완화가 안정성에 도움되는가"부터 저비용 검증.
2. 도움이 된다는 신호가 나오면 Revise 모드 엔진(§5) 구현 착수 — rl 모드 재설계 때와 동일하게 유닛 테스트 + 스모크 테스트(2만 스텝) 선행, 본 학습은 그 이후.
3. 구현/실측 완료 후 `design_spec.md`에 정식 편입 (현재는 이 문서에만 존재하는 설계 단계).
