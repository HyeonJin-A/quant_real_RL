# <span style="color: #FFFFFF;">V9 Design</span>

V8까지 이어온 수작업 선형 스코어링(가중합 Score) 체계를 폐기하고, **진입과 청산을 모두 강화학습(RL) 정책이 직접 결정하는 풀 컨트롤(Full-Control) 아키텍처**로 전환하는 V9 설계서입니다. V8 최적화 결과가 보여준 "극소 거래 횟수 + 승률 ~12% + 단 1건의 대박 수익이 전체 PnL을 지배"하는 과적합/체리픽 현상을 근본적으로 해결하는 것이 목표이며, 이를 위해 **BTC + ETH 멀티심볼 학습**과 **사전 확정 합격 기준 기반의 엄격한 검증 프로토콜**을 도입합니다.

*(본 버전의 코드는 기존 코드를 수정하지 않고 `src/rl_v9/` 신규 패키지로 전부 새로 작성합니다. 기존 코드는 로직 의미 확인용 레퍼런스로만 사용합니다.)*

---

## <span style="color: #FFA500;">1. 개요 및 배경 (V8의 한계와 V9의 결정)</span>

### <span style="color: #2E8B57;">V8이 남긴 문제</span>
- **거래 횟수 부족**: CMA-ES 100,000회 최적화로 수익 파라미터를 찾았으나, 거래 횟수가 극단적으로 적어 통계적 신뢰가 없습니다.
- **한방 전략 잔존**: 표준편차 페널티에도 불구하고 실제 Top1 시뮬레이션은 다수의 -20 USDT 손실 속에 단 1건의 +1,206 USDT 수익이 전체를 지배하는 구조였습니다 (승률 ~12%).
- **선형 스코어의 표현력 한계**: 8개 피처의 min-max 정규화 가중합이라는 선형 구조로는 피처 간 상호작용(예: "ADX가 낮을 때만 RSI 다이버전스가 유효")을 표현할 수 없어, 파라미터 튜닝으로 짜낼 수 있는 성능의 천장에 도달했습니다.

### <span style="color: #2E8B57;">V9의 핵심 결정</span>
1. **스코어 공식 폐기 → RL 정책으로 대체**: 진입 방향/여부/시점과 청산 시점을 모두 PPO 정책망이 학습합니다. `threshold`, 피처 가중치, 반익절/본절/손절 레벨 등 V8의 전략 파라미터는 전부 사라집니다.
2. **룰 로직의 역할 축소**: 기존 파동 감지 엔진(`find_dynamic_pivots`)은 **피처 공급자**로만 남습니다. 매매 의사결정에는 관여하지 않습니다.
3. **멀티심볼 학습**: BTC-USDT-SWAP 단독 학습에서 벗어나 ETH-USDT-SWAP을 학습 데이터에 추가하여, 한 종목의 특수 패턴에 과적합되지 않은 일반화 정책을 강제합니다.
4. **V8 Score는 평가 전용으로 격상**: V8의 통합 Score 공식(MSL 페널티, 거래수 보너스, 표준편차 페널티)은 보상 함수가 아닌 **검증셋 모델 선택 및 최종 평가의 채점 기준**으로만 사용합니다.

## <span style="color: #FFA500;">2. 아키텍처 개요</span>

### <span style="color: #2E8B57;">신규 패키지 구조 (`src/rl_v9/`)</span>
```
src/rl_v9/
├── prep_features_v9.py   # 심볼 파라미터화된 피처 캐시 생성 (버그 수정판)
├── env_v9.py             # 풀 컨트롤 Gymnasium 트레이딩 환경
├── train_v9.py           # PPO 학습 엔트리포인트 (SubprocVecEnv + 검증 콜백)
└── eval_v9.py            # 평가 하네스 (룰베이스 V8 대비, 합격 기준 자동 판정)
```

### <span style="color: #2E8B57;">데이터 흐름</span>
```
{BTC, ETH}-USDT-SWAP 1m/5m CSV
    ↓  prep_features_v9.py (파동 감지 + 피처 계산, 심볼별)
features_v9_{SYMBOL}.npy  (1m 행 단위 인과적(causal) 피처 캐시)
    ↓  env_v9.py (정규화된 ~18차원 관측 벡터)
PPO 정책망 (64×64 MLP)
    ↓  행동: Hold / Open-Long / Open-Short / Close
env 내장 시장 메커니즘 (수수료, 고정 사이징, 강제청산가)
    ↓  보상: 매 스텝 mark-to-market 손익 변화량
학습 → 검증셋 V8 Score 기반 체크포인트 선택 → 테스트셋 최종 1회 평가
```

### <span style="color: #2E8B57;">유지 / 대체 / 폐기 요소</span>
| 요소 | V8 | V9 |
|---|---|---|
| 파동/피봇 감지 (`find_dynamic_pivots`) | 피처 생성 | **유지** (피처 공급 전용) |
| 선형 가중합 Score + threshold | 진입/스위칭 판단 | **폐기** → RL 정책 |
| 반익절/본절이동/ATR 손절 | 하드코딩 청산 룰 | **폐기** → RL이 청산 학습 |
| 즉각 스위칭 (S&R) | 룰 (반대 score ≥ threshold) | **폐기** → 보유 중 반대 방향 Open = 플립으로 학습 |
| 0.382 Pre-hit 필터 | 진입 취소 룰 | **폐기** → 관측 피처로 전환 (정책이 필터를 학습) |
| 고정 사이징 100 USDT × 20배 | 유지 | **유지** (V8과 동일한 손익 회계) |
| 수수료 0.05% (진입/청산 각각) | 유지 | **유지** |
| Optuna CMA-ES | 파라미터 탐색 | **폐기** (탐색할 전략 파라미터가 없음) |
| V8 통합 Score 공식 | 최적화 목적함수 | **평가/모델 선택 전용으로 전환** |

## <span style="color: #FFA500;">3. 피처 파이프라인 (캐시 재빌드)</span>

### <span style="color: #2E8B57;">캐시 재생성 원칙</span>
- `prep_features_v9.py`는 기존 v7 스크립트(`scratch/prep_features_v7.py`)의 인과성(causality) 보장 구조를 계승하되, **심볼을 파라미터로 받아** `src/rl_v9/caches/features_v9_{SYMBOL}.npy`를 생성합니다. (v7은 BTC 경로가 하드코딩되어 있었음 — prep_features_v7.py:40-45)
- 5m 파동을 1m 스냅샷에 매핑하는 방식은 V8과 동일: 매 1m 행은 "그 시점에 실시간으로 알 수 있었던" 마지막 파동의 피처만 담습니다.
- V8 dtype에서 미사용 컬럼(VWAP, POC, SR 라인 6종)은 제거하고, **`volume` 기반 상대강도 컬럼을 실제 계산값으로 채웁니다** (아래 버그 수정 참조).

### <span style="color: #2E8B57;">정규화 규칙 (누출 금지 원칙)</span>
- 모든 관측 피처는 **사전에 손으로 확정한 고정 상수**(클립 범위 + 스케일 변환)로 정규화합니다.
- **데이터셋 통계(퍼센타일, 평균/표준편차 등)를 정규화에 사용하는 것을 금지**합니다. 기존 env가 전체 10년 데이터의 99퍼센타일로 ATR을 정규화하던 방식은 (1) 미래 데이터가 과거 관측에 스며드는 train/test 누출이고 (2) 라이브 환경에서 재현 불가능하므로 폐기합니다.
- 고정 상수는 V8 설계의 피처 범위(wave_scale 1~6%, duration 0.02~3.0일, vol_ratio 1~4 등)를 출발점으로 하되, **4장의 심볼별 분포 리포트로 BTC/ETH 양쪽을 커버하는지 검증 후 확정**하고 문서/코드에 상수로 기록합니다.
- 꼬리가 긴 피처(duration, vol_ratio 등)는 `log1p` 후 클립, 바운디드 지표(RSI, ADX)는 `/100`.

## <span style="color: #FFA500;">4. 멀티심볼 데이터 (BTC + ETH) 및 전처리 정의</span>

### <span style="color: #2E8B57;">전처리 필요성 검토 결론</span>
두 종목은 가격 스케일(BTC ~10만 USDT vs ETH ~수천 USDT)과 변동성이 다르지만, **피처 대부분이 이미 스케일 불변(%·비율·바운디드 지표)이라 필요한 전처리는 경량**입니다. 감사 결과:

| 항목 | 스케일 의존성 | 처리 |
|---|---|---|
| wave_scale_percent (%), wave_duration_day (일) | 불변 | 그대로 사용 |
| RSI (0~100), ADX (0~100) | 불변 | 그대로 사용 |
| divergence_price_gap_percent (%) | 불변 | 그대로 사용 |
| volatility_ratio (캔들폭 / 직전 10캔들 평균폭) | 불변 | 그대로 사용 |
| relative_volume_strength (자기 이력 500캔들 내 min-max 백분위) | 불변 | 그대로 사용 (종목별 거래량 단위 차이가 비율 계산에서 소거됨) |
| 피보나치 되돌림 위치 (파동 내 상대 위치) | 불변 | 그대로 사용 |
| **원시 ATR** (BTC 수백 USDT vs ETH 수 USDT) | **의존** 🚨 | **`atr / close` 비율(%)로 변환** — 종목 간 비교 가능 + 시계열 비정상성(가격 10배 상승 시 ATR 드리프트)도 동시 해결 |
| 보상/손익 회계 | 불변 | `pos_size = (100 × leverage) / price`로 가격이 소거되어, 같은 %변동이면 종목 무관 동일한 USDT 손익 |

### <span style="color: #2E8B57;">피처 분포 검증 절차 (상수 확정 전 필수)</span>
- ETH는 BTC보다 %변동성이 커서 wave_scale, vol_ratio 등의 **분포 자체가 오른쪽으로 치우칠 수 있습니다**.
- `prep_features_v9.py`는 캐시 생성 시 **심볼별 피처 분포 리포트**(피처별 1/5/25/50/75/95/99 퍼센타일 표)를 함께 출력합니다.
- 고정 정규화 클립 범위는 이 리포트를 근거로 **두 종목 분포를 모두 커버하도록** 확정합니다. (예: BTC 기준 wave_scale 클립 상한 6%가 ETH 95퍼센타일을 잘라먹으면 상한을 상향)

### <span style="color: #2E8B57;">멀티심볼 학습 구성</span>
- **Env는 심볼 1개 데이터셋**을 받는 단순 구조를 유지하고, 병렬 환경(VecEnv) 워커를 심볼별로 배분합니다 (예: 8워커 = BTC 4 + ETH 4). 에피소드는 각 심볼 데이터 내 랜덤 시작.
- **관측에 심볼 ID를 넣지 않습니다.** 정책이 "지금 BTC인지 ETH인지"를 모르게 하여 종목 특수 패턴 암기를 차단하고, 두 종목에서 동시에 통하는 일반화 패턴만 학습하도록 강제합니다. (심볼 one-hot 추가는 로드맵의 ablation 항목)
- **데이터 분할은 날짜 기준 동일 경계**로 심볼별 70/15/15 — 테스트 구간이 두 종목에서 같은 기간이 되도록 하여, 특정 종목만 유리한 장세를 배정받는 왜곡을 방지합니다. (ETH 1m 데이터: 2019-12-25 ~ 현재, 약 6.5년 — BTC와 거의 동일 구간)
- **평가는 심볼별 개별 지표 + 합산 지표를 모두 산출**하며, 8장의 사전 합격 기준은 **종목별로 각각** 적용합니다. 한 종목의 수익이 다른 종목의 손실을 가리는 것을 방지합니다.

## <span style="color: #FFA500;">5. 환경 설계 (Env — 풀 컨트롤)</span>

### <span style="color: #2E8B57;">결정 주기 및 에피소드 구조</span>
- **결정 주기: 매 1m 행** (V8의 1분 스냅샷 시뮬레이션과 동일한 시장 해상도). 빠른 실험용으로 `decision_stride` 설정(기본 1, 실험 시 5)을 둡니다.
- **에피소드: 랜덤 시작 + 고정 길이 청크** (14~30일 = 20,160~43,200 스텝, 학습 초기엔 짧게 시작하는 커리큘럼 허용). 청크 끝은 `truncated=True`로 처리해 가치함수 부트스트랩이 올바르게 동작하게 합니다.
- 기존 env의 "전체 히스토리 = 단일 에피소드, 고정 시작점" 구조는 샘플 다양성이 없어 폐기합니다.
- 에피소드 종료 시 보유 포지션은 종가 강제 청산으로 정산합니다.

### <span style="color: #2E8B57;">관측 공간 (Observation, ~18차원 단일 스텝 벡터)</span>
| 그룹 | 피처 | 비고 |
|---|---|---|
| 파동 피처 (8) | wave_scale, wave_duration, 방향조정 RSI (`rsi if bullish else 100-rsi`), div_gap, volatility_ratio, **relative_volume_strength (수정판)**, ADX, **atr/close** | 캐시에서 로드, 고정 상수 정규화 |
| 파동 문맥 (5) | **is_bullish (±1)** 🚨, 피보나치 되돌림 위치, 0.382 pre-hit 플래그, 파동 경과시간(log1p), 되돌림 위치 델타(5m/15m) | pre-hit "필터"는 룰이 아니라 관측 정보로 제공 — 쓸지 말지는 정책이 학습 |
| 포지션 상태 (4~5) | 포지션 방향 (-1/0/+1), 미실현손익 % (클립), 보유 시간(log1p), 강제청산가까지의 거리 % | 청산 학습에 필수 |

- 🚨 기존 env는 `is_bullish`를 로드만 하고 관측에 넣지 않는 결함이 있었습니다. 파동 방향은 셋업의 정체성이므로 반드시 포함합니다.
- 10스텝 윈도우(82차원) 구조는 폐기합니다. 파동 피처는 저속 변화라 윈도우가 중복 정보였고, 시간 변화는 델타 피처 2종으로 명시적으로 공급합니다.
- 관측 행렬은 env 초기화 시 **전체를 float32로 사전 정규화**해 두고, 스텝마다 행 슬라이스만 수행합니다 (기존 per-step 파이썬 루프의 병목 제거).

### <span style="color: #2E8B57;">행동 공간 (Action, Discrete(4))</span>
- `0: Hold` — 무포지션이면 대기, 보유 중이면 유지
- `1: Open-Long` — 무포지션이면 롱 진입. **숏 보유 중이면 즉시 청산 후 롱 진입 (플립)** = V8 스위칭의 학습 버전
- `2: Open-Short` — 위의 대칭
- `3: Close` — 보유 포지션 전량 청산. 무포지션이면 no-op (Hold와 동일 처리)
- V8의 반익절(50% 부분 청산)은 V9.0에서 제외합니다. 부분 청산 행동을 추가하면 행동 공간과 회계가 복잡해지므로 로드맵 항목으로 미룹니다.

### <span style="color: #2E8B57;">시장 메커니즘 (env에 내장되는 것은 "시장의 물리 법칙"만)</span>
- **고정 사이징**: 매 진입 100.0 USDT 증거금 × 레버리지 20배, `pos_size = 2000 / price`. 복리·자본금 개념 없음 (V8과 동일한 손익 회계 → 룰베이스와 직접 비교 가능).
- **수수료**: 진입/청산 각각 notional의 0.05% 차감. 플립은 청산 수수료 + 진입 수수료가 모두 발생.
- **강제청산 (Liquidation)**: 미실현손실이 증거금(100 USDT)에 도달하는 가격에서 포지션 강제 종료 — 이는 전략 룰이 아니라 거래소 메커니즘이므로 env에 남습니다. 단일 거래 최대 손실이 자연히 약 -100 USDT로 상한됩니다.
- 기존 env의 `initial_balance=1000` + `balance ≤ 0` 파산 종료 + 고정 사이징의 삼중 모순 구조는 폐기합니다. 잔고는 info 딕셔너리의 누적 통계로만 기록합니다.

## <span style="color: #FFA500;">6. 보상 설계 (Reward)</span>

### <span style="color: #2E8B57;">기본 보상: 매 스텝 Mark-to-Market 손익 변화량 (Dense Reward)</span>
```
reward_t = (equity_t − equity_{t−1}) / 100.0
equity = 실현 누적손익 + 미실현손익 − 발생 수수료
```
- **채택 이유**: 풀 컨트롤 환경의 최대 난관은 "수백~수천 스텝 뒤 청산 시점에 몰아서 받는 지연 보상"의 credit assignment입니다. 매 스텝 평가손익 변화를 즉시 보상으로 주면 보유 결정 하나하나가 즉각 피드백을 받아 학습이 안정됩니다. 에피소드 총보상은 실현 손익 합과 일치하므로 목적 왜곡이 없습니다.
- **스케일**: 증거금 100 USDT로 나눠 스텝 보상을 대략 [-0.1, +0.1], 거래 단위 누적으로 [-1, +3] 수준에 놓습니다 (PPO 가치손실 안정 범위). `VecNormalize` 보상 정규화는 재현성을 해치므로 사용하지 않습니다.
- **수수료는 발생 스텝에서 즉시 음(-)의 보상**: 과잉 거래(churning)가 자연 억제됩니다. 별도의 인위적 홀딩 페널티(-0.01/스텝 등)는 두지 않습니다.

### <span style="color: #2E8B57;">할인율: γ = 0.999</span>
- 1m 스텝 기준 γ=0.99는 유효 horizon ~100분에 불과해 수 시간~수일짜리 포지션 운용을 학습할 수 없습니다. γ=0.999(유효 horizon ~16시간)를 기본값으로 하고, 필요 시 0.9995까지 실험합니다.

### <span style="color: #2E8B57;">V8 Score는 보상에 넣지 않는다 (원칙)</span>
- MSL×5 페널티, 거래수 보너스, 표준편차 페널티는 **에피소드 전체가 끝나야 계산되는 궤적 단위 통계**라 스텝 단위 credit assignment을 오염시키고, 거래수 보너스는 보상에 넣는 순간 churning 인센티브가 됩니다.
- 따라서 **보상은 순손익만 단순하게** 유지하고, V8 Score는 7장의 검증 콜백에서 **체크포인트 선택 기준**으로, 8장의 최종 평가에서 **채점 기준**으로만 사용합니다. ("단순한 보상, 엄격한 평가" 원칙)
- 최대 단일 손실 억제는 보상이 아니라 강제청산 메커니즘(구조적 상한 -100 USDT)이 담당합니다.

## <span style="color: #FFA500;">9. 실패 시나리오와 Fallback (사전 결정 게이트)</span>

풀 컨트롤 트레이딩 RL의 전형적 실패 모드와 대응 경로를 **학습 시작 전에** 확정합니다:

| 실패 신호 (10M 스텝 시점 판정) | 진단 | Fallback 경로 |
|---|---|---|
| 거래를 거의 안 함 (검증 롤아웃 월 1회 미만) | "아무것도 안 하기"가 국소최적 (수수료 회피) | ent_coef 상향/감쇠 완화 → 그래도 실패 시 **Fallback A** |
| 진입 직후 즉시 청산 반복 | 청산 학습이 진입 학습을 방해 | **Fallback A: 하이브리드** — ATR 하드 손절만 룰로 복원, RL은 진입 + 익절/조기청산만 담당 |
| 보유만 하고 청산을 안 함 | 청산 credit assignment 실패 | 최대 보유시간 강제 청산 추가 → 실패 시 **Fallback A** |
| 하이브리드도 실패 | 풀 컨트롤 자체가 과욕 | **Fallback B: 진입만 학습** — V8 청산 엔진(반익절/BE/ATR-SL)을 그대로 재사용하고 RL은 진입 판단만 (semi-MDP 구조, 학습 난이도 최저) |

각 Fallback은 env 설정 변경으로 전환 가능하도록 `env_v9.py`에 `exit_mode: {"rl", "hybrid", "rule"}` 옵션을 설계 단계부터 포함합니다.

2. **ATR 정규화의 전체 데이터 99퍼센타일 사용** (v8_trading_env.py — train/test 누출 + 가격 비정상성): → `atr/close` 비율 + 고정 상수 정규화.
3. **`is_bullish` 관측 누락** (로드만 하고 obs 미포함): → 신규 관측에 포함.
4. **사이징/잔고/파산의 삼중 모순** (고정 100불 사이징인데 복리 잔고 추적 + `balance≤0` 종료): → 고정 사이징 + 강제청산가 메커니즘으로 일원화, 파산 개념 삭제.
5. **backtest_rl.py 승률 계산 스텁** (win/loss 카운트가 `pass`로 방치되어 항상 0): → 거래 단위 기록 기반의 `eval_v9.py`로 대체.
6. **requirements.txt에 RL 의존성 미고정** (SB3/gymnasium/torch/optuna가 venv에만 존재): → 버전 핀 추가.
7. **Const.py 15m 라이브 설정과 V8 설계서 불일치** (threshold 70 vs 60, leverage 25 vs 20): V9에서는 해당 파라미터 자체가 사라지므로 자연 해소되나, 라이브 전환 전까지 기존 데모가 설계서와 다른 파라미터로 구동 중임을 여기에 기록해 둡니다.
